# Как обучен детектор wake word «Эй, кот» — полный разбор

Документ для передачи: что за задача, какие данные готовили, как устроено обучение,
на какие грабли наступили и что в итоге получилось. Самодостаточный — контекст
проекта знать не нужно.

---

## 1. Задача

Голосовая активация ассистента по кодовой фразе — как «Алиса» у Яндекс.Станции.
Микрофон слушает постоянно; как только звучит **«Эй, кот»** — срабатывает триггер.

Формально это **keyword spotting (KWS)**: бинарная классификация «фраза произнесена / нет»
на скользящем окне аудио. Не распознавание речи (STT) — модель не знает слов, она
знает один акустический паттерн.

**Требования, которые формировали решение:**

- **Полностью локально** — на маке (Apple Silicon), без облака, приватно.
- **Детектор фразы, а не голоса** — срабатывает на «Эй, кот» от *любого* человека.
  Speaker-specific вариант (только голос владельца) мы пробовали и осознанно отвергли.
- **Дёшево на инференсе** — крутится фоном постоянно, не должен греть процессор.
- Низкая частота ложных срабатываний важнее последних процентов recall: детектор,
  который сам себя запускает от случайной речи, бесполезен.

## 2. Архитектура

```
аудио 16 кГц
   └─> melspectrogram (ONNX, предобучен)        ← заморожен
        └─> Google speech embedding (ONNX)      ← заморожен
             └─> тензор (16 кадров, 96)
                  └─> Conv1D-голова             ← ОБУЧАЕМ ТОЛЬКО ЭТО
                       └─> sigmoid → скор 0..1
```

Основа — предобученные модели из [openWakeWord](https://github.com/dscripka/openWakeWord):
мел-спектрограмма и Google speech embedding. Они **заморожены**, мы их не трогали.
Это стандартный приём для KWS: тяжёлое представление речи берём готовым, а под
конкретную фразу учим крошечный классификатор поверх.

**Геометрия:** окно **2 секунды** аудио → эмбеддинг **(16 кадров × 96 признаков)**.

**Обучаемая голова** (рецепт из разбора [hey-gaia](https://aryanranderiya.com/agent-convos/hey-gaia-wake-word-trained/)):

```
(B, 16, 96) → transpose → (B, 96, 16)
  Conv1d(96 → 64, kernel=3, padding=1) → BatchNorm → ReLU → Dropout
  Conv1d(64 → 64, kernel=3, padding=1) → BatchNorm → ReLU
  AdaptiveAvgPool1d(1) → Linear(64 → 1) → Sigmoid
```

~30K параметров, экспорт в ONNX — **~200 КБ**.

**Почему именно свёртка по времени.** Полносвязную голову (FCN) мы пробовали первой —
она провалилась. Conv1D сворачивает **по оси времени** и потому ловит не «мешок звуков»,
а *паттерн перехода* «эй» → «кот». Для фразы из двух слогов это решающе.

## 3. Данные

Всё, кроме реальной речи и записи фона, синтезировано — живых людей не записывали.

| Класс | Источник | Объём |
|---|---|---|
| **Позитивы** | ElevenLabs TTS, ~70 русских голосов × варианты фразы | ~1000 клипов → **22.5k окон** |
| Негатив: **реальная речь** | корпус Golos (Сбер), `bond005/sberdevices_golos_10h_crowd` | ~20k окон |
| Негатив: **похожие слова** | ElevenLabs: «эй код», «эй крот», «эй бот»… (вес ×4) | ~500 клипов |
| Негатив: **шум** | синтетика: белый/розовый, тоны, чирпы, клики | ~2200 окон |
| Негатив: **эфир** | запись реального фона микрофона | ~285 окон |

### Позитивы

Генерируются `gen_dataset.py` через ElevenLabs (модель `eleven_multilingual_v2`).
Варианты фразы: `«Эй, кот»`, `«Эй, кот!»`, `«Эй кот»`, `«Эй, кот.»`, `«Эй-эй, кот»` —
разная просодия и паузы.

**Аугментация — 25 вариантов на клип** (`augment_positive` в `wake_common.py`):

- темп и высота ±10% (`resample_poly` со случайным отношением);
- реверберация с вероятностью 0.4 — синтетический импульсный отклик «комната/микрофон»;
- случайный сдвиг позиции фразы внутри 2-секундного окна;
- аддитивный шум со случайным SNR (8–30 дБ);
- случайная громкость (0.6–1.1).

Без аугментации модель переобучается на «стерильный» TTS и разваливается на живом
микрофоне: реальная комната даёт реверб, а живая речь — разный темп.

### Негативы — где основная работа

Recall на фразе достигается легко. Вся сложность — **не срабатывать на всё остальное**.
Негативы добавлялись итеративно, каждый слой закрывал конкретный наблюдаемый провал:

1. **Реальная русская речь (Golos)** — главный рычаг. Именно она убирает ложняки на
   обычный разговор рядом с микрофоном. Аналог LibriSpeech в англоязычных рецептах.
2. **Похожие слова** (`gen_negatives.py`) — «эй код», «эй крот», «эй бот» и т.п.
   Идут с **весом ×4** в лоссе: это самые опасные негативы, фонетически почти позитивы.
3. **Синтетический шум** — добавлен после того, как ранняя версия срабатывала на
   стук, шелест и тоны.
4. **Реальный эфир** — записанный фон комнаты и микрофона. Убрал ложняки на «тишину»,
   которая на деле не тишина, а собственный шум тракта.

Класс негативов режется скользящим окном 2 с с шагом 0.5 с — из одной длинной реплики
получается много обучающих окон.

### Что осознанно НЕ делали

- **Не записывали голос владельца в обучение.** Была промежуточная версия
  speaker-specific (40 записей владельца как позитивы, чужие «эй кот» — как негативы).
  Отвергли: она даёт противоречивый сигнал (одна и та же фраза и позитив, и негатив)
  и сужает применимость. Записи владельца ушли в **held-out** для честной проверки.

## 4. Пайплайн обучения

Намеренно **разделён на два скрипта**, чтобы ошибка в обучении не стоила пересчёта
дорогих признаков:

### Шаг 1 — `build_cache.py` (долго, один раз, ~10 минут)

Считает эмбеддинги всех групп данных и складывает в `out/cache/feat_*.npy`
(`feat_pos`, `feat_neg_rand`, `feat_neg_hard`, `feat_noise`, `feat_amb`).

Оптимизации:
- **распараллелено по ядрам** — `ThreadPoolExecutor`, у каждого потока **своя**
  `AudioFeatures` (ONNX-сессии нельзя шарить между потоками);
- порядок чанков сохраняется (`ex.map`), поэтому результат воспроизводим;
- обработка **чанками по 128 окон** — иначе построение всех ~360k окон разом даёт OOM.

### Шаг 2 — `train_local.py` (быстро, ~30 секунд)

Грузит готовый кэш, обучает голову, экспортирует ONNX.

- устройство — **MPS** (GPU Apple), 70 эпох, **cosine LR**;
- **взвешенный BCE**: баланс классов (позитивы редкие) + **×4** на похожие слова;
- **SpecAugment** — случайное маскирование кадров и каналов, векторизованно, без
  Python-цикла;
- экспорт: `torch.onnx.export(..., opset_version=13, dynamo=False,
  dynamic_axes={"features": {0: "batch"}})`.

Разделение окупилось многократно: архитектуру головы, веса классов и порог можно
крутить сколько угодно, каждый прогон — полминуты.

### Отдельно — `embed_speech.py`

Скачивает Golos потоково и считает из него фичи-негативы в `out/neg_speech_feats.npy`.
Тонкость: с `datasets` 5.0 нужно `Audio(decode=False)` и ручной декод через
`soundfile` (иначе тянется torchcodec и ломается).

## 5. Два урока, которые стоили больше всего времени

### 🔴 Урок 1: признаки должны быть ДЕТЕРМИНИРОВАНЫ

**Это был корневой баг, испортивший почти всю работу.**

Батчевый метод openWakeWord `F.embed_clips` из-за использования `np.empty`
(неинициализированная память) возвращал **разные признаки при каждом вызове для одного
и того же аудио** — разброс порядка 20 при диапазоне значений ±60.

Последствия: и обучение, и детект работали на случайном мусоре. Модель вела себя
необъяснимо — «иногда ловит, чаще нет», причём не реагировала даже на те записи,
которые использовались как позитивы.

**Фикс:** везде использовать только `F._get_embeddings` (потоковый путь,
детерминирован, расхождение между вызовами 0). Функция `embed_all` в `wake_common.py`
собрана на нём: мел-спектрограмма по клипу + один вызов эмбеддинга.

**Организационный вывод:** проверять модель нужно через **реальный путь инференса**
(`detect.py`), а не только по внутренней метрике обучения. Иначе рассинхрон
train/inference остаётся невидимым — метрики при обучении выглядели прилично, пока
живой детектор молчал.

### ⚠️ Урок 2: обучать надо ТАМ ЖЕ, где инференс

Изначально пытались тренировать в Google Colab (в том числе headless через
`google-colab-cli`). Модель обучалась нормально, но локально давала **скор 0 на всё**.

Причина: `melspectrogram`-ONNX считает признаки **по-разному на x86 (Colab) и ARM
(Apple Silicon)** — корреляция всего **0.79** для одного и того же файла. Дело не в
GPU/CPU, не в numpy и не в версии onnxruntime — сама реализация FFT в ONNX расходится
между архитектурами.

**Вывод:** признаки и обучение — на той же машине, где будет инференс. Для этой задачи
Colab оказался и не нужен: голова крошечная, обучение занимает полминуты локально.

## 6. Результаты

Честный held-out: **150 голосов, не участвовавших в обучении** (голоса с индексом ≥ 60
исключены из train), плюс отложенная часть реальной речи Golos.

| Порог | Recall (невиданные голоса) | FP на похожих словах | FP на реальной речи |
|---|---|---|---|
| 0.70 | 98.8% | 0.0% | 0.07% |
| **0.85** | **98.5%** | **0.0%** | **0.07%** |
| 0.90 | 98.0% | 0.0% | 0.07% |

Проверка через реальный путь детекта (`_get_embeddings` → ONNX): позитивы **14/14 = 1.00**,
стабильно между запусками.

Для сравнения, эталон hey-gaia: 99% recall / 0.4% FP — по ложным срабатываниям
получилось лучше.

Обработка одного окна — **~45 мс**, реакция детектора — **0.3–0.5 с**.

## 7. Инференс: два предохранителя

Сама модель — только половина дела. В `detect.py` поверх неё два механизма:

1. **Энергетический гейт** — если пиковая амплитуда окна ниже `GATE` (по умолчанию 1200),
   скор вообще не считается, сразу 0. Убрало ложняки на тишине и сэкономило CPU.
2. **Сглаживание `CONSEC`** — срабатывание засчитывается, только если порог держится
   `CONSEC` окон подряд (по умолчанию 2). Режет одиночные случайные всплески.

Рабочая конфигурация: `THRESHOLD=0.9 GATE=1200 CONSEC=2`.

Компромисс скорость/надёжность регулируется шагом окна `HOP`: 4000 сэмплов = проверка
раз в 0.25 с; 2000 = раз в 0.125 с (быстрее реакция, вдвое больше CPU — при обработке
45 мс на окно запас есть).

## 9. Воспроизведение

**Окружение:** Python 3.12 в `wake/.venv`:
`openwakeword` (**git-версия**, `pip install -e . --no-deps` — релиз с PyPI отличается),
`torch`, `onnx`, `onnxruntime`, `sounddevice`, `scipy`, `numpy`, `datasets`,
`soundfile`, `scikit-learn`. Ключ ElevenLabs — в `wake/.env` (в git не коммитится).

```bash
# 1. данные (опционально — если нужно пересобрать)
VOICES=70   .venv/bin/python gen_dataset.py       # позитивы «Эй, кот»
VOICES=12   .venv/bin/python gen_negatives.py     # чужая речь + похожие слова
TARGET=25000 .venv/bin/python embed_speech.py     # реальная речь (Golos)

# 2. кэш признаков — долго, ~10 мин, только при смене данных
.venv/bin/python build_cache.py

# 3. обучение головы — ~30 с, можно гонять сколько угодно
.venv/bin/python train_local.py

# 4. проверка живьём
THRESHOLD=0.9 GATE=1200 CONSEC=2 .venv/bin/python detect.py
```

При скачивании с HuggingFace нужен `HF_HUB_DISABLE_XET=1` — иначе загрузка виснет
на `cas-bridge.xethub.hf.co`.

## 10. Что можно улучшить дальше

Recall на фразе фактически решён (~99%). Основной фронт — дальнейшее снижение ложных
срабатываний на реальной речи:

- **больше реальной речи в негативах** — главный рычаг; в эталонных рецептах счёт идёт
  на сотни тысяч окон, у нас ~20k;
- расширение набора похожих слов (фонетические соседи фразы);
- тонкая настройка порога и `CONSEC` под конкретное помещение и микрофон.
