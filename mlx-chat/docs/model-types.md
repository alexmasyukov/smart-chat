# Типы моделей: для каждой задачи — своя модель

Главная мысль: **не бывает «лучшей модели вообще».** Трансформеры делятся на три
архитектурных типа, и под каждую задачу подходит свой. Взять не тот тип — как
забивать гвозди микроскопом: вроде работает, но медленно, дорого и глупо.

## Три архитектурных типа

### 1. Энкодеры (encoder-only) — «понимальщики»

Семейство **BERT**. Русские: **ruBERT, ruBERT-tiny2, ruRoBERTa, ruSBERT** (Сбер / комьюнити).

- **Обучение:** masked language modeling — прячут слова в тексте, модель угадывает
  их, **видя контекст с обеих сторон** (bidirectional). Читает весь вход целиком и осмысляет.
- **Сильны в:** классификация, NER (извлечение сущностей), эмбеддинги, поиск,
  похожесть, реранкинг.
- **Не генерируют текст** — выдают метку класса или вектор.

### 2. Декодеры (decoder-only) — «генераторы»

Семейство **GPT**. Русские/используемые: **ruGPT3, YandexGPT, GigaChat, LFM2.5,
Qwen, Llama, GPT-5, Claude**.

- **Обучение:** next-token prediction — предсказывают **следующий токен** слева
  направо (causal, видят только прошлое). Пишут по одному слову.
- **Сильны в:** генерация, чат, рассуждение, следование инструкциям, вызов
  инструментов, код. Универсалы.

### 3. Энкодер-декодеры (seq2seq) — «переводчики»

Семейство **T5 / BART**. Русские: **FRED-T5, ruT5, mT5**.

- **Обучение:** text-to-text — энкодер читает вход, декодер генерирует выход.
- **Сильны в:** перевод, суммаризация, перефраз, исправление текста — «один текст → другой текст».

## Сводная таблица

| Тип | Направление | Обучение | Сильны в | Русские модели |
|---|---|---|---|---|
| **Энкодер** | двусторонний | угадать спрятанное | классификация, поиск, эмбеддинги, NER | ruBERT(-tiny2), ruRoBERTa, ruSBERT |
| **Декодер** | слева направо | следующий токен | генерация, чат, агенты, код | ruGPT3, YandexGPT, GigaChat, LFM2.5 |
| **Энкодер-декодер** | вход → выход | текст → текст | перевод, суммаризация, перефраз | FRED-T5, ruT5, mT5 |

## Практические правила («задача → тип модели»)

| Задача | Тип | Пример |
|---|---|---|
| Разложить по полкам (класс из набора) | **Энкодер** | роутинг команд, интенты, тональность, спам, модерация |
| Найти похожее / поиск / реранк | **Энкодер** (эмбеддер) | семантический поиск, RAG-retrieval, дедуп |
| Извлечь сущности из текста | **Энкодер** | имена, даты, адреса, номера |
| Написать / поговорить / порассуждать | **Декодер** | чат-ассистент, генерация текста, код |
| Выбрать и вызвать инструмент свободно | **Декодер** | агент, tool-calling |
| Переделать один текст в другой | **Энкодер-декодер** | перевод, суммаризация, нормализация |

Ещё легче классификации: иногда хватает **эмбеддер + логистическая регрессия**
(даже без дообучения энкодера целиком) — но дообученный мелкий энкодер с головой
обычно точнее и почти так же прост.

## Почему это важно — наш живой пример

Мы решали **классификацию** команд («открой папку X» → имя из 5 вариантов) и сперва
взяли **декодеры** (генераторы) — потому что они универсальны и «на слуху». Результат
после подбора и дообучения:

| | ruBERT-tiny2 (**энкодер**, 29M) | LFM2.5-350M (**декодер**, LoRA) | LFM2.5-1.2B (декодер, промпт) |
|---|---|---|---|
| Точность | 80/80 + 20/20 новых | 80/80 + 20/20 новых | 17/17 (кое-как, день подбора) |
| Скорость | **~1.5 мс (CPU)** | ~26 мс (GPU) | ~330 мс |
| Размер | **29M** | 350M | 1.2B |

**Правильный тип (энкодер) в 29M сделал то же, что декодер в 40 раз больше.** Вот
и вся мораль: сначала пойми ТИП задачи — потом бери модель под него, а не «побольше и универсальнее».

## Ещё одна ось: размер под ширину задачи

Внутри выбранного типа: **чем уже и стабильнее задача — тем мельче модель, если её
дообучить.** Узкий классификатор → крошечный энкодер. Широкий «понимай всё и рассуждай»
→ большой декодер (или API). Дообучение бьёт промпт на узких задачах; промпт/RAG
выигрывают на широких и меняющихся.
