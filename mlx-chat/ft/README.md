# ft — классификатор команд «открыть папку»

Локальный классификатор: запрос пользователя → имя команды (или `none`).
Команда возвращается как имя; открытие папки — на стороне приложения.

Есть **две реализации** одной задачи (обе обучаются локально на своих данных):

| | **ruBERT-tiny2** (энкодер) | **LFM2.5-350M** (декодер, LoRA) |
|---|---|---|
| Параметры | **29M** | 350M |
| Точность (test80) | 80/80 | 80/80 |
| Новые формулировки (compare) | 20/20 | 20/20 |
| Скорость (прогрето) | **~1.5 мс / запрос (CPU)** ≈ ~650 класс/сек | ~26 мс (MLX/GPU) |
| RAM | ~0.12 ГБ | ~0.58 ГБ |
| Язык | русский родной (Сбер) | многоязычная |
| Системный промпт | не нужен (голый текст) | нужен короткий |
| Обучение | ~30 сек | ~2 мин |
| Запуск в LM Studio | нет (нужна Python-служба) | **да** |
| Тулчейн | transformers + torch | mlx-lm + Outlines |

Вывод эксперимента: **на узкой классификации в закрытый набор мелкий энкодер
(29M) бьёт генеративный декодер кратно большего размера** — та же точность и
обобщение, в ~17× быстрее, в 12× меньше, на голом CPU. Выбор между ними — это
выбор «отдельная быстрая служба» (ruBERT) vs «крутится прямо в LM Studio» (350M).

## Данные

Источник формулировок — **`phrases.txt`** (человекочитаемый банк фраз, редактируй его).
`gen_dataset.py` читает его и добавляет к каждой фразе вариации: **опечатки, разный
регистр, лишние слова, перестановку слов**. Итог:

- `data/train.jsonl`, `data/valid.jsonl` — chat-формат (для LFM2.5-350M);
- `data/preview.md` — весь датасет по-человечески, по классам (для просмотра глазами).

Класс `none` обязателен: без него модель на «привет» или «открой youtube»
вынуждена выбрать одну из папок. Он учит отвечать «ничего не подходит».

## Как обучить / проверить

**Правишь набор:** команды — в `../classifier.py` (`TOOLS`), формулировки — в `phrases.txt`.

**ruBERT-tiny2 (энкодер, рекомендуется как самый быстрый):**
```bash
cd mlx-chat && HF_HUB_DISABLE_XET=1 .venv/bin/python ft/train_bert.py
# обучает + оценивает на held-out, сохраняет в ft/bert_model/
```

**LFM2.5-350M (декодер, для LM Studio):**
```bash
cd mlx-chat && ./ft/train.sh
# gen_dataset -> LoRA -> fuse -> eval; модель в ft/fused/
```

**Проверки:**
```bash
.venv/bin/python ft/test_bert_full.py  # ruBERT по всем наборам + проверка утечки в train
.venv/bin/python ft/test80.py     # 80 кейсов на дообученной 350M
.venv/bin/python ft/compare.py    # ruBERT-29M vs LFM-350M на свежих фразах + скорость
.venv/bin/python ft/eval.py --adapter ft/adapters   # held-out для 350M
```

## Ключевые грабли (проверено на практике)

- **Дообученной генеративной модели (350M) Structured Output/грамматику надо
  ВЫКЛЮЧАТЬ.** Она обучена выдавать голое имя; JSON-схема навязывает незнакомый
  формат → сваливается в дефолт (`open_adsw` на всё) И тормозит вдвое.
  Грамматика (Outlines / LM Studio Structured Output) нужна только НЕобученной модели.
- **В LM Studio для 350M:** temperature `0`, repeat penalty off, Structured Output off,
  системный промпт — короткий (см. `SHORT_SYSTEM` в `../classifier.py`).
- **ruBERT** системного промпта не использует вообще — подаётся голый текст запроса.
- Скачивание с HF виснет на Xet → всегда `HF_HUB_DISABLE_XET=1`.

## Файлы

- `phrases.txt` — банк фраз (источник истины по формулировкам).
- `gen_dataset.py` — генерация датасета + `data/preview.md`.
- `train.sh` — пайплайн LFM2.5-350M (LoRA → fuse → eval).
- `train_bert.py` — обучение ruBERT-tiny2 (энкодер).
- `eval.py` — held-out для 350M (Outlines-грамматика).
- `test80.py` — 80 кейсов (свободная генерация, как в LM Studio).
- `compare.py` — ruBERT vs 350M + замер скорости.
- `test_bert_full.py` — ruBERT по всем трём наборам сразу; сначала проверяет, что
  каждый набор реально held-out (нет в обучающих данных), потом меряет точность.
- `data/` — датасет и preview (мелкое, в git идёт).
- `adapters/`, `fused/`, `bert_model/` — веса (в git НЕ идут).
