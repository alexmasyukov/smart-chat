# ft — разбор голосовых команд

Запрос пользователя → команда и её параметры. Вход — **расшифровка Whisper** из
голосового ассистента (`my-pro/assistant`), поэтому со всеми искажениями живой
речи. Что открывать и как — решает приложение, модель только разбирает.

**В бою:** `tiny_model` — своя модель, обученная с нуля, 467K параметров,
1.8 МБ, 0.69 мс на запрос. Поднимается через `assistant/scripts/router_server.py`
(HTTP на 51790).

```
«открой адсв на ветке ард 2511»  → open_adsw          {ticket: ард, num: 2511}
«Открой папку ADSV.»             → open_adsw_folder   {}
«Девнетворк открой»              → open_network_dev   {}
«плохо распознает номер ветки»   → (пусто — не команда)
```

## Что распознаётся

| Интент | Действие | Слоты |
|---|---|---|
| `open_adsw`, `open_network` | стенд фича-ветки | `ticket`, `num`, `branch` |
| `open_adsw_folder`, `open_network_folder` | папка в Finder | — |
| `open_components` | `/Users/alex/arenadata-ui` | — |
| `open_projects` | `/Users/alex/my-pro` | — |
| `open_adsw_prod` | `https://adsw.io/` | — |
| `open_network_prod` | `https://network.arenadata.io/` | — |
| `open_adsw_dev` | `https://int.dev.plt.adsw.io/` | — |
| `open_network_dev` | `https://network.ext.dev.plt.adsw.io/` | — |
| (пустой ответ) | не команда | — |

**Слоты:** `ticket` — буквенный ключ (`ард`, `ARD`, `ит дев`), `num` — номер,
`branch` — имя ветки словами (`main`, `про логин`, `feature/x`), `target` — чем
открыть (`в браузере`, `в идее`).

`ticket` и `num` раздельно, чтобы приложение собрало имя по своей конвенции.
Дефолтный префикс, когда назван только номер, тоже ставит приложение: модель
сообщает лишь то, что реально прозвучало.

**Стенд vs папка** различаются формулировкой: «папку», «директорию»,
«в проводнике», «в finder» → `_folder`. Без них — стенд.

## История: три модели на одной задаче

Задачу решали трижды, и каждый раз модель уменьшалась:

| | LFM2.5-350M | ruBERT-tiny2 | **с нуля** |
|---|---|---|---|
| Параметров | 350M | 29.2M | **467K** |
| Размер | ~400 МБ | 114 МБ | **1.8 МБ** |
| Скорость | ~26 мс | 1.2 мс | **0.69 мс** |
| Предобучение | многоязычное | русский корпус | **нет** |

**Вывод, ради которого всё это делалось.** На узкой задаче в закрытый набор
предобучение почти не работает: 92% веса ruBERT — таблица словаря на 83 828
токенов, из которых нужны около трёх тысяч слов. Модель, обученная только на
наших данных, не уступает ей ни на одном наборе и **лучше обобщает** на
незнакомых словах — посимвольный вход устойчивее к искажениям Whisper, чем
словарный (подробности в разделе «Модель с нуля»).

Старые ветки экспериментов: `lfm-json` (350M с JSON-выходом, заморожена),
`multi-command` (составные команды на ruBERT).

## Данные

**Правишь `templates.txt`** — это источник истины. Там шаблоны со слотами по
интентам плюс банки значений; генератор сам подставляет значения и потому знает
границы слотов, руками ничего не размечаем.

Секции файла:

| Секция | Что |
|---|---|
| `# <intent>` | шаблоны фраз для интента |
| `# slot:<name>` | значения слота (`ticket`, `branch`, `target`) |
| `# alias:<intent>` | написания названия проекта, включая ошибки Whisper |
| `# bank:<name>` | служебные слова (`folder`, `explorer`, `prod`, `dev`) — тег `O` |
| `# service` | «фича-ветку» и прочее: тип ветки, а не имя |

Подстановки в шаблонах: `{NNNN}` — случайный номер, `{WORD}` / `{SLUG}` —
случайное имя ветки, `{branch}` / `{target}` — слоты, остальные `{имя}` берутся
из банков и получают тег `O`.

`phrases.txt` — старый банк плоских фраз, тоже подмешивается. Его **не правим**:
он кормит `gen_dataset.py` для 350M. Писался он для классификатора папок, где
`open_adsw` означал каталог, поэтому `gen_slots.py` перечитывает его — фразы со
словом «папку» уезжают в `*_folder`, остальные остаются стендами.

Пустой ответ (бывший `none`) обязателен: без него модель на «привет» вынуждена
выбрать какую-то команду. Отдельно нужны длинные не-команды со словами
«ветка»/«открой» — на них модель однажды открывала стенд посреди разговора.

## Как обучить / проверить

```bash
cd mlx-chat
.venv/bin/python ft/gen_slots.py              # посмотреть датасет и разметку
.venv/bin/python ft/train_tiny.py             # обучить (2 мин 20 с на MPS)
.venv/bin/python ft/test_tiny.py              # все наборы разом
```

Настройки обучения через окружение: `DEVICE=cpu`, `SEED=7`, `EPOCHS=60`.

После переобучения **перезапустить сервер** — он держит веса в памяти и сам их
не перечитает:

```bash
lsof -nP -iTCP:51790 -sTCP:LISTEN -t | xargs -r kill
HF_HUB_DISABLE_XET=1 nohup .venv/bin/python \
  ~/my-pro/assistant/scripts/router_server.py > /tmp/router.log 2>&1 &
```

**Легаси-модели** (в бою не используются, оставлены для сравнения):

## Слоты (joint-модель): интент + параметры одной моделью

«открой в браузере адсв на фиче ветке ард 1120» →
`open_adsw` + `{target: "в браузере", ticket: "ард", num: "1120"}`.

Та же ruBERT-tiny2, но **две головы поверх одного энкодера**: интент по `[CLS]`,
слоты — BIO-теги по токенам. Форвард один, поэтому скорость та же (~1.1 мс).
Слот-голова стоит 312×9 ≈ 2.8К параметров.

**Результат:** held-out 36/36 (интент и слоты), живые расшифровки Whisper 17/17,
~1.1 мс (CPU). Вытаскивает и невиданные имена веток («на ветке зюзюки») — то,
что регуляркой не взять; ради этого всё и делалось.

**Вход — речь, а не текст.** Потребитель (`my-pro/assistant`) кормит модель
расшифровкой Whisper, отсюда требования в `assistant/MODEL_REQUESTS.md`: точка
в конце почти всегда, названия коверкаются (`adsw` → «ADSV», «АДСВ», **«Адрес
свой»**), латиница мешается с кириллицей, слова склеиваются («фичеветку»,
«Fitch-оветку»), порядок свободный. Всё это есть в обучающих данных —
`test_whisper.py` гоняет 17 живых расшифровок из боевого лога.

### Слоты

| Слот | Что | Пример |
|---|---|---|
| `ticket` | буквенный ключ тикета, любое написание | `ард`, `АРД`, `ard`, `ARD` |
| `num` | номер тикета, только цифры | `7777` |
| `branch` | произвольное имя ветки целиком (когда номера нет) | `про логин`, `main`, `feature/x` |
| `target` | чем открыть | `в браузере`, `в идее` |

`ticket` и `num` — **раздельные слоты**, чтобы приложение собрало имя по своей
конвенции (`ард` + `7777` → `ARD-7777`). Голый номер тоже ловится: «на фиче 999»
даёт `num` без `ticket` — префикс подставляет приложение.

**Данные — `templates.txt`** (`phrases.txt` не трогали, он кормит 350M и
`train_bert.py`). Шаблоны со слотами: генератор сам подставляет значения и
потому знает границы — руками не размечаем. Старые плоские фразы подтягиваются
с авторазметкой `target` (там уже есть «открой adsw в finder»).

Ключевое: **номера и имена подставляются случайно, а не берутся из списка.**
Проверено вживую: пока словесные имена веток брались из фиксированного банка,
модель заучивала сами слова и на «ветке коли» (нет в банке) отдавала пустой
слот. Со случайными значениями — включая бессмысленные псевдослова вроде
«зюзюки» — учится позиция: «после „на ветке“ стоит имя, каким бы оно ни было».

Формы записи тикета (через пробел / через дефис / голый номер) комбинирует
`gen_slots.py` из банка `slot:ticket` — в самом банке лежат только префиксы.

Использование (боевая модель):
```python
import tiny_model_def as tm
model, cfg = tm.load("ft/tiny_model")

cmds = tm.predict(model, cfg, "открой адсв на ветке ард 7777")
# [{"intent": "open_adsw", "slots": {"ticket": "ард", "num": "7777"},
#   "score": 0.9998}]

for c in cmds:
    if c["score"] < 0.7:        # порог на стороне потребителя
        continue
    s = c["slots"]
    # сборка имени ветки — сторона приложения, конвенция своя
    branch = f"ARD-{s['num']}" if "num" in s else s.get("branch")
```

**Ответ — список команд**, пустой значит «не команда» (бывший `none`). Сейчас
там всегда ноль или одна: составные фразы из обучения убраны, но контракт
оставлен, чтобы не ломать потребителя дважды.

У легаси-модели `joint_model_def` контракт тот же, только `load()` и `predict()`
дополнительно принимают токенизатор.

`score` — усреднённая по токенам команды уверенность интента. Учти: модель
обучена до почти нулевого лосса на узком домене, поэтому уверена и когда
ошибается — порог отсекает не всё. По слотам вероятность не возвращается.

При интенте `none` слот `branch` не выдаётся: на посторонней речи это не имя
ветки, а случайное слово («плохо распознаёт номер ветки» → `branch: номер`),
и потребитель читал бы его как настоящее.

Модель отдаёт значения **кусками исходного текста** («ард», «7777»). Привести их
к `ARD-7777` для `git checkout` — дело приложения: конвенцию именования модель
знать не может, она видела только запрос.

**Разбор целиком делает модель, постобработки нет.** Текст режется на буквы,
цифры и знаки препинания по отдельности — одинаково в обучении (`gen_slots.seg`)
и в `predict()`. Поэтому «ARD-2020.» приходит как четыре токена `ARD`, `-`,
`2020`, `.`, и модель сама ставит им `B-ticket`, `O`, `B-num`, `O`. Слитное
«ARD2020» разделяет то же правило — граница буквы и цифры. Раньше здесь стояли
регулярка разбора ID и снятие точки постфактум; с правильной сегментацией они
просто не нужны.

Значение слота собирается срезом исходной строки по offsets, а не склейкой
токенов — иначе «feature/new-header» пришлось бы угадывать, где были пробелы.

## Модель с нуля (467K) — без предобучения вообще

Веса стартуют случайными, всё знание — из наших ~12 500 сгенерированных
примеров. Повод: задача узкая — **около 3000 уникальных слов, 78 символов,
максимум 18 слов в запросе**, а решали её энкодером на 29M, предобученным на
всём русском интернете.

Устройство — классический char-BiLSTM тэггер:

```
символы слова → char-BiLSTM ──┐
                              ├─→ word-BiLSTM ─┬─→ голова интента (по токенам)
слово → словарный эмбеддинг ──┘                └─→ голова слотов (по токенам)
```

**Обе головы потокенные, интент размечается тем же BIO, что и слоты.** Это
позволяет фразе распадаться на несколько команд: «открой нетворк на 2070 и адсв
на 3511» → два span'а, и каждый слот достаётся своей команде. При интенте на всю
фразу привязать номер к проекту было бы нечем. `B-` отделяет и две соседние
команды с одинаковым интентом.

Составные фразы сейчас **из обучения убраны** (в употреблении не нужны, а длина
фразы из-за них росла с 18 слов до 42 и вдвое удорожала обучение). Архитектура
их поддерживает: `MULTI_COUNT` в `gen_slots.py` и переобучить. Контракт ответа
остаётся списком команд — сейчас в нём ноль или одна.

Посимвольный вход — не экономия, а суть: на входе расшифровки Whisper, где
слова коверкаются. Модель, читающая буквы, видит «Натворк» и «нетворк»
соседями; словарная — двумя разными токенами. Словарный эмбеддинг оставлен для
частых слов («открой», «ветке»), где важно само слово, а не его форма. В
словарь берём слова от 2 вхождений: 2244 слова из 3032 встречаются ровно раз —
это случайные значения слотов, запоминать их незачем.

**Результат на тех же наборах:**

| | С нуля (467K) | ruBERT (29.2M) |
|---|---|---|
| Живые Whisper | **57/57** | (мерялось на 17: 17/17) |
| Одиночные held-out | 35/36 | 35/36 |
| Размер | **1.8 МБ** | 114 МБ |
| Скорость ответа | **0.69 мс** | 1.20 мс |
| Обучение | 2 мин 20 с (MPS) | ~1 мин |

Раньше, до упаковки последовательностей, модель была медленнее ruBERT (1.43 мс
против 1.07). Сейчас вдвое быстрее — лишний паддинг убран.

Устойчивость проверялась на 5 сидах (на меньшем наборе интентов): интент и живые
расшифровки — идеально во всех прогонах, слоты — 36/36 в четырёх из пяти.

**Обобщение на невиданных словах — лучше, чем у предобученной:**

| Запрос | С нуля | ruBERT |
|---|---|---|
| «на ветке гребешка» | `branch: гребешка` | слот потерян |
| «на ветке коли» | `branch: коли` | слот потерян |
| «release/2024-q3» | точно | развалила на `release/2024-` + `num: q` |
| «про оплату картой» | весь span | обрезала до «про оплату» |
| «какая погода в москве» | `none`, слотов нет | `none`, но выдумала `target: в москве` |

Последняя строка — суть: **предобучение навредило**. ruBERT знает, что «в
браузере» — предложный оборот, и перенесла это на «в москве». Модель с нуля
такого предрассудка не имеет.

выкинуть питон-сервис из ассистента — реальная задача. Две LSTM и два линейных
слоя переносятся руками.

## Ключевые грабли (проверено на практике)

- **Дообученной генеративной модели (350M) Structured Output/грамматику надо
  ВЫКЛЮЧАТЬ.** Она обучена выдавать голое имя; JSON-схема навязывает незнакомый
  формат → сваливается в дефолт (`open_adsw` на всё) И тормозит вдвое.
  Грамматика (Outlines / LM Studio Structured Output) нужна только НЕобученной модели.
- **В LM Studio для 350M:** temperature `0`, repeat penalty off, Structured Output off,
  системный промпт — короткий (см. `SHORT_SYSTEM` в `../classifier.py`).
- **ruBERT** системного промпта не использует вообще — подаётся голый текст запроса.
- Скачивание с HF виснет на Xet → всегда `HF_HUB_DISABLE_XET=1`.
- **Слоты: на инференсе резать текст ровно как при обучении** (та же `seg()` +
  `is_split_into_words=True`). Любой рассинхрон сегментации бьёт по метрике:
  когда обучали на `text.split()`, а на инференсе резали токенизатором по
  пунктуации, `ARD-4242` обрывался до `ARD` — минус 30%.
- **Сегментация решает, нужна ли постобработка.** Пока слово резалось только по
  пробелам, `ARD-2020.` был одним токеном с одним тегом, и части приходилось
  доставать регуляркой после модели. Разрезали на буквы/цифры/знаки — надобность
  в регулярке исчезла сама. Костыль после модели почти всегда означает, что
  что-то не так с подготовкой входа.
- **Пунктуация как отдельные токены удорожает слот-голову.** Доля тега `O`
  выросла, сигнала стало относительно меньше — `SLOT_WEIGHT` пришлось поднять с
  2.0 до 4.0 и добавить эпох, иначе номер после префикса терялся.
- **Мерить, а не предполагать: MPS быстрее CPU даже на мелкой LSTM.** Ожидание
  было обратным («LSTM на MPS плохо оптимизирован, модель крошечная, накладные
  расходы съедят выигрыш») — по факту 36 с против 4 мин 16 с, в 7 раз.
- **Один прогон ничего не доказывает.** И у ruBERT, и у модели с нуля слабое
  место переезжает между запусками: то теряется одиночное имя ветки, то
  обрывается длинный слот. Пока обучение стоит полминуты, гонять 5 сидов —
  единственный способ отличить свойство модели от везения. Подгонка параметров
  под один упавший кейс — это не починка, а подгонка под тест.
- **BiLSTM обязана быть упакованной (`pack_padded_sequence`).** Обратный проход
  стартует с паддинга и тащит его состояние в настоящие слова, поэтому результат
  зависел от того, до какой длины добит батч, а обучение и применение были
  обязаны паддить одинаково. Попытка урезать паддинг ради скорости мгновенно
  сломала распознавание не-команд. С упаковкой паддинг не влияет вовсе — и это
  дало и правильность, и скорость (2.37 мс → 0.67 мс).
- **Паддинг батча идёт по самой длинной фразе — следи за хвостом.** Составные
  команды подняли максимум с 18 слов до 42, и обучение подорожало вдвое: char-RNN
  считает «батч × ширина» слов. Сортировка по длине и паддинг по ширине батча
  вернули среднюю ширину к 6.6 слов.
- **Слитные формы надо прописывать буквально.** Шаблон `{prod}адсв` даёт ДВА
  токена, а Whisper произносит это одним словом — «Протодсв», «Девнетворк».
  Пока таких форм не было в данных, они работали случайно и рассыпались, как
  только число интентов выросло.
- **Мерить, а не предполагать (снова).** Я решил, что 10-минутное обучение —
  плата за упаковку, и ошибся: упаковка стоила 23 мс на шаг, а виноват был
  паддинг до 42 слов. Замер занял минуту и указал на другое место.
- **Слоты: тег читать с первого subword слова, хвост игнорировать.** У хвоста при
  обучении стоит `-100`, его предсказание — мусор: `dev` → `['de','##v']` отдавал
  span `de`. Ещё 30% метрики.
- **Аугментация не должна вставлять филлер между `ticket` и `num`.** Получалось
  «ард пожалуйста 574» — живые люди так не говорят, а модель учится рвать пару.
- **Следить за балансом внутри слота.** Когда тикеты заняли половину позиций
  `{branch}`, примеров с произвольными именами стало меньше и «на ветке васи»
  начало теряться. Лечится долей (сейчас 58% имён) и объёмом.
- **Обучать на том, что реально приходит на вход.** Модель не видела пунктуации
  — и на боевом «ARD2020**.**» номер отваливался: регулярка разбора ID не
  переживала точку. Данные должны повторять вход, а не идеализировать его.
- **Ошибки распознавания — часть данных.** Whisper устойчиво слышит `adsw` как
  «Адрес свой»; лечится не постобработкой, а банком алиасов (`# alias:<intent>`
  в `templates.txt`), который подставляется в каждый шаблон.

## Потребитель

Ассистент `my-pro/assistant` ходит в модель по HTTP через
`scripts/router_server.py` (порт 51790), маппинг ответа — в `LocalRouter.swift`.

Два документа связывают проекты:

- `assistant/MODEL_REQUESTS.md` — что просят доработать в модели (пишет ассистент);
- `assistant/MODEL_CHANGES.md` — что изменилось и что править в приложении.

**Свежие расшифровки** берутся из `~/Library/Logs/Assistant/commands.jsonl` —
угадать, как Whisper исковеркает слово, нельзя, можно только увидеть.
