# ft — классификатор команд «открыть папку»

Локальный классификатор: запрос пользователя → имя команды (или `none`).
Команда возвращается как имя; открытие папки — на стороне приложения.

Есть также **joint-модель** — та же ruBERT, но вдобавок вытаскивает параметры
из запроса (ветку и чем открыть). См. раздел «Слоты» ниже.

Есть **две реализации** одной задачи (обе обучаются локально на своих данных):

| | **ruBERT-tiny2** (энкодер) | **LFM2.5-350M** (декодер, LoRA) |
|---|---|---|
| Параметры | **29M** | 350M |
| Точность (test80) | 80/80 | 80/80 |
| Новые формулировки (compare) | 20/20 | 20/20 |
| Скорость (прогрето) | **~1.5 мс / запрос (CPU)** ≈ ~650 класс/сек | ~26 мс (MLX/GPU) |
| RAM | ~0.12 ГБ | ~0.58 ГБ |
| Язык | русский родной (Сбер) | многоязычная |
| Системный промпт | не нужен (голый текст) | нужен короткий |
| Обучение | ~30 сек | ~2 мин |
| Запуск в LM Studio | нет (нужна Python-служба) | **да** |
| Тулчейн | transformers + torch | mlx-lm + Outlines |

Вывод эксперимента: **на узкой классификации в закрытый набор мелкий энкодер
(29M) бьёт генеративный декодер кратно большего размера** — та же точность и
обобщение, в ~17× быстрее, в 12× меньше, на голом CPU. Выбор между ними — это
выбор «отдельная быстрая служба» (ruBERT) vs «крутится прямо в LM Studio» (350M).

Карта имя→папка — в `../classifier.py` (`COMMANDS`).

## Данные

Источник формулировок — **`phrases.txt`** (человекочитаемый банк фраз, редактируй его).
`gen_dataset.py` читает его и добавляет к каждой фразе вариации: **опечатки, разный
регистр, лишние слова, перестановку слов**. Итог:

- `data/train.jsonl`, `data/valid.jsonl` — chat-формат (для LFM2.5-350M);
- `data/preview.md` — весь датасет по-человечески, по классам (для просмотра глазами).

Класс `none` обязателен: без него модель на «привет» или «открой youtube»
вынуждена выбрать одну из папок. Он учит отвечать «ничего не подходит».

## Как обучить / проверить

**Правишь набор:** команды — в `../classifier.py` (`TOOLS`), формулировки — в `phrases.txt`.

**ruBERT-tiny2 (энкодер, рекомендуется как самый быстрый):**
```bash
cd mlx-chat && HF_HUB_DISABLE_XET=1 .venv/bin/python ft/train_bert.py
# обучает + оценивает на held-out, сохраняет в ft/bert_model/
```

**LFM2.5-350M (декодер, для LM Studio):**
```bash
cd mlx-chat && ./ft/train.sh
# gen_dataset -> LoRA -> fuse -> eval; модель в ft/fused/
```

**Проверки:**
```bash
.venv/bin/python ft/test_bert_full.py  # ruBERT по всем наборам + проверка утечки в train
.venv/bin/python ft/test80.py     # 80 кейсов на дообученной 350M
.venv/bin/python ft/compare.py    # ruBERT-29M vs LFM-350M на свежих фразах + скорость
.venv/bin/python ft/eval.py --adapter ft/adapters   # held-out для 350M
```

## Слоты (joint-модель): интент + параметры одной моделью

«открой в браузере адсв на фиче ветке ард 1120» →
`open_adsw` + `{target: "в браузере", ticket: "ард", num: "1120"}`.

Та же ruBERT-tiny2, но **две головы поверх одного энкодера**: интент по `[CLS]`,
слоты — BIO-теги по токенам. Форвард один, поэтому скорость та же (~1.1 мс).
Слот-голова стоит 312×9 ≈ 2.8К параметров.

**Результат (held-out, 36 кейсов):** интент 36/36, слоты 36/36, ~1.1 мс (CPU).
Вытаскивает и невиданные имена веток («на ветке зюзюки») — то, что регуляркой
не взять; ради этого всё и делалось.

### Слоты

| Слот | Что | Пример |
|---|---|---|
| `ticket` | буквенный ключ тикета, любое написание | `ард`, `АРД`, `ard`, `ARD` |
| `num` | номер тикета, только цифры | `7777` |
| `branch` | произвольное имя ветки целиком (когда номера нет) | `про логин`, `main`, `feature/x` |
| `target` | чем открыть | `в браузере`, `в идее` |

`ticket` и `num` — **раздельные слоты**, чтобы приложение собрало имя по своей
конвенции (`ард` + `7777` → `ARD-7777`). Голый номер тоже ловится: «на фиче 999»
даёт `num` без `ticket` — префикс подставляет приложение.

**Данные — `templates.txt`** (`phrases.txt` не трогали, он кормит 350M и
`train_bert.py`). Шаблоны со слотами: генератор сам подставляет значения и
потому знает границы — руками не размечаем. Старые плоские фразы подтягиваются
с авторазметкой `target` (там уже есть «открой adsw в finder»).

Ключевое: **номера и имена подставляются случайно, а не берутся из списка.**
Проверено вживую: пока словесные имена веток брались из фиксированного банка,
модель заучивала сами слова и на «ветке коли» (нет в банке) отдавала пустой
слот. Со случайными значениями — включая бессмысленные псевдослова вроде
«зюзюки» — учится позиция: «после „на ветке“ стоит имя, каким бы оно ни было».

Формы записи тикета (через пробел / через дефис / голый номер) комбинирует
`gen_slots.py` из банка `slot:ticket` — в самом банке лежат только префиксы.

```bash
cd mlx-chat
.venv/bin/python ft/gen_slots.py --preview   # датасет + data/slots_preview.md
.venv/bin/python ft/train_joint.py           # обучить (~1 мин), в ft/joint_model/
.venv/bin/python ft/test_joint.py            # held-out: интент и слоты отдельно
```

Использование:
```python
import joint_model_def as jm
model, tok, cfg = jm.load("ft/joint_model")

intent, slots = jm.predict(model, tok, cfg, "открой адсв на ветке ард 7777")
# ("open_adsw", {"ticket": "ард", "num": "7777"})

# сборка имени ветки — сторона приложения, конвенция своя
branch = f"ARD-{slots['num']}" if "num" in slots else slots.get("branch")
# "ARD-7777"
```

Модель отдаёт значения **кусками исходного текста** («ард», «7777»). Привести их
к `ARD-7777` для `git checkout` — дело приложения: конвенцию именования модель
знать не может, она видела только запрос.

Одно исключение, где всё же работает код: `ARD-4242` написан **одним словом**, а
теги ставятся по словам — разделить разметкой физически нельзя. Такие случаи
модель метит как `ticket`, а `predict()` разбирает регуляркой
`^([А-Яа-яA-Za-z]+)[-_ ]?(\d+)$`. Это разбор канонического идентификатора, а не
извлечение параметра из живой речи — всю работу по речи («ард 7777» → два
разных слота) делает модель.

## Ключевые грабли (проверено на практике)

- **Дообученной генеративной модели (350M) Structured Output/грамматику надо
  ВЫКЛЮЧАТЬ.** Она обучена выдавать голое имя; JSON-схема навязывает незнакомый
  формат → сваливается в дефолт (`open_adsw` на всё) И тормозит вдвое.
  Грамматика (Outlines / LM Studio Structured Output) нужна только НЕобученной модели.
- **В LM Studio для 350M:** temperature `0`, repeat penalty off, Structured Output off,
  системный промпт — короткий (см. `SHORT_SYSTEM` в `../classifier.py`).
- **ruBERT** системного промпта не использует вообще — подаётся голый текст запроса.
- Скачивание с HF виснет на Xet → всегда `HF_HUB_DISABLE_XET=1`.
- **Слоты: на инференсе резать текст ровно как при обучении** (`text.split()` +
  `is_split_into_words=True`). Обычная токенизация рвёт `ARD-4242` по пунктуации
  на три слова, а обучалась модель на нём как на одном — теги у «-» и «4242»
  оказываются необученным мусором, и слот обрывается до `ARD`. Стоило 30% метрики.
- **Слоты: тег читать с первого subword слова, хвост игнорировать.** У хвоста при
  обучении стоит `-100`, его предсказание — мусор: `dev` → `['de','##v']` отдавал
  span `de`. Ещё 30% метрики.
- **Аугментация не должна вставлять филлер между `ticket` и `num`.** Получалось
  «ард пожалуйста 574» — живые люди так не говорят, а модель учится рвать пару.
- **Следить за балансом внутри слота.** Когда тикеты заняли половину позиций
  `{branch}`, примеров с произвольными именами стало меньше и «на ветке васи»
  начало теряться. Лечится долей (сейчас 58% имён) и объёмом.

## Файлы

- `phrases.txt` — банк фраз (источник истины по формулировкам).
- `gen_dataset.py` — генерация датасета + `data/preview.md`.
- `train.sh` — пайплайн LFM2.5-350M (LoRA → fuse → eval).
- `train_bert.py` — обучение ruBERT-tiny2 (энкодер).
- `eval.py` — held-out для 350M (Outlines-грамматика).
- `test80.py` — 80 кейсов (свободная генерация, как в LM Studio).
- `compare.py` — ruBERT vs 350M + замер скорости.
- `test_bert_full.py` — ruBERT по всем трём наборам сразу; сначала проверяет, что
  каждый набор реально held-out (нет в обучающих данных), потом меряет точность.
- `data/` — датасет и preview (мелкое, в git идёт).
- `adapters/`, `fused/`, `bert_model/`, `joint_model/` — веса (в git НЕ идут).

Joint-модель (интент + слоты):

- `templates.txt` — шаблоны со слотами и банки значений (источник истины).
- `gen_slots.py` — генерация размеченных данных (BIO), `--preview` для глаз.
- `joint_model_def.py` — модель (энкодер + две головы), `load()` / `predict()`.
- `train_joint.py` — обучение joint-модели.
- `test_joint.py` — held-out: интент и слоты меряются отдельно.
