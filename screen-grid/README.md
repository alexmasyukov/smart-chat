# screen-grid

Детектор границы цветового блока на экране: сканирует одну горизонтальную
строку слева направо и ищет, где цвет меняется. Никакого таймера — детект
запускается вручную, кнопкой.

Инфраструктура (два отдельных macOS-приложения + Python-детектор) не привязана
к конкретному алгоритму — она уже пережила несколько итераций алгоритма и,
скорее всего, переживёт ещё.

## Части

- **`detect.py`** — Python HTTP-сервер (`:8132`). По запросу `GET /scan`:
  снимает экран (`screencapture`), сканирует, публикует результат, пишет лог.
  Никакого фонового цикла — только по требованию.
- **`overlay.swift`** → бинарник `overlay` — прозрачное click-through окно на
  весь экран. Рисует точки, их номера и границу блока (long-poll к `/points`,
  перерисовывается сразу, как только детектор опубликовал новый скан).
- **`button.swift`** → бинарник `button` — обычное macOS-окно с заголовком:
  два ползунка (глубина уточнения границы, шаг сетки), большая серая кнопка
  «Scan» и таблица лога справа (номер / hex-цвет / кружок-превью).
- **`run.sh`** — собирает оба бинарника (если исходники новее) и запускает
  детектор + оба приложения.

```
./run.sh
```

## Алгоритм

Работает по одной горизонтальной строке `y = ROW_FRAC * H` (по умолчанию
центр экрана), в пределах левых `REGION_FRAC` экрана (по умолчанию 30%).

### 1. Сетка

От `X_OFFSET` с шагом `step` (ползунок «Шаг сетки», по умолчанию 150px,
диапазон 10–300) ставим точки `0, 1, 2, ...` и читаем цвет в каждой —
**срединный (медианный)** цвет патча `PATCH×PATCH` пикселей (медиана, а не
среднее: одиночный выброс-пиксель не утягивает цвет точки). Сравнение
цветов **точное** (по округлённому hex-значению), без допуска на похожесть.

### 2. Блок = прогон ≥ `MIN_RUN` точек одного цвета

Граница **не** ставится по первой же сломанной точке — иначе получались бы
блоки в 10px шириной, которые не нужны. **Блок = прогон из `MIN_RUN` точек
подряд одного цвета** (по умолчанию 3). Идём слева направо, растим прогон;
на смене цвета прогон закрывается и начинается новый.

Отсюда прямое следствие (как и хотелось): если точка `0` отличается от
`1, 2, 3, 4`, а `1..4` одного цвета — блок начинается с точки `1`, а точка
`0` это **край/шум** и отбрасывается (её прогон короче `MIN_RUN`).

### 3. Вертикальный столбик проб (фильтр текста внутри блока)

Смена цвета в точке `j` может быть не границей, а **текстом** внутри блока.
Чтобы отличить, ставим у `j` **вертикальный столбик проб** — `VPROBES` точек
вверх и столько же вниз (по умолчанию 2+2) с **тем же шагом** `step`. Шаг
большой — пробы уходят далеко по вертикали, мимо строки текста, в чистый фон
блока. Голосуем цветом проб против цвета текущего блока:

- **большинство** проб цвета блока — значит `j` это текст: лечим цвет `j` и
  **не рвём** прогон, блок продолжается;
- иначе — цвет действительно сменился, прогон закрывается.

Пробы столбика — настоящие точки (`kind="vprobe"`, на оверлее **голубые**),
они видны и попадают в лог: наглядно, на что опирался детектор.

> Вертикальный шаг = горизонтальному. Для высоких панелей это надёжно, но
> у **низкого** блока (кнопка ~40px) проба на `±2·step` уедет за блок и
> соврёт — это осознанный компромисс выбранного варианта.

**Граница** = правый край **первого** блока (прогона ≥ `MIN_RUN`), но только
если дальше есть **ещё** блок ≥ `MIN_RUN` — то есть граница именно **между
двумя настоящими блоками**, а не сползание в шум или край региона. Найдя её,
дальше не идём.

### 4. Уточнение границы (бисекция)

Между `i` (ещё старый цвет) и `i+1` (уже другой) находим точку ровно
посередине — **проверочную точку** ("probe", на оверлее красится в другой
цвет, оранжевый). Смотрим её цвет:

- совпал со старым (`run_color`) — блок продолжается дальше вправо,
  новый отрезок для деления: `(эта точка .. i+1)`;
- не совпал — граница левее, новый отрезок: `(i .. эта точка)`.

Повторяем — ползунок «Уточнений границы» (`bisect`, по умолчанию 3,
диапазон 1–10) задаёт, сколько раз. Останавливаемся раньше, если отрезок
сузился до `BISECT_MIN_GAP` (5px) — дальше мельчить бессмысленно.

Последняя точка, которая ЕЩЁ дала старый цвет — и есть уточнённая граница
блока. По ней рисуется вертикальная линия на весь экран.

### 5. Нумерация и порядок в логе

Номер точки = порядок **обнаружения**: сначала сетка получает номера
`0..n-1` слева направо, дальше каждая новая проверочная точка получает
следующий свободный номер по мере нахождения (`n`, `n+1`, ...).

Но в массиве точек, на оверлее и в логе точки стоят по своему **месту на
экране** (геометрическая позиция), а не в порядке присвоения номера. То
есть проверочная точка №7, найденная между точками 3 и 4, физически стоит
между ними: `0, 1, 2, 3, 7, 4, 5, 6` — так удобнее сопоставлять с картинкой
на экране.

## Протокол между детектором и приложениями

- `GET /scan?bisect=N&step=M` — один проход сканирования с этими
  параметрами (иначе берутся дефолты `BISECT_MAX`/`STEP`). Отвечает
  `{ok, count, v, numbers, colors}`.
- `GET /points?since=V` — long-poll: висит, пока не появится скан новее
  `V`, отвечает сразу как только он готов. Поле `v` — версия, растёт с
  каждым сканом; `points`/`colors`/`kinds`/`numbers`/`lines` — данные для
  отрисовки.
- `GET /health` — жив ли детектор.

## Вывод на диск (`out/`)

- `out/screenshot.png` — последний снимок экрана, который анализировал
  детектор (специально сохраняется В `out/`, не во временную папку).
- `out/points_<метка_времени>.txt` — лог КАЖДОГО скана, отдельный файл,
  формат `номер-hex` по одной точке на строку, в геометрическом порядке.
- `out/preview.png` — только у `python3 detect.py --snapshot` (отладочный
  разовый прогон с отрисовкой поверх скриншота, без запуска сервера).

## Настройка через переменные окружения (детектор)

| Переменная | Дефолт | Что |
|---|---|---|
| `SG_PORT` | `8132` | порт HTTP-сервера |
| `SG_REGION_FRAC` | `0.30` | какую долю ширины экрана слева сканируем |
| `SG_STEP` | `150` | шаг сетки, px (переопределяется `?step=` за запрос) |
| `SG_X_OFFSET` | `10` | отступ первой точки от левого края |
| `SG_ROW_FRAC` | `0.5` | какую строку сканируем (0..1 по высоте) |
| `SG_PATCH` | `6` | размер патча для медианного цвета точки, px |
| `SG_VPROBES` | `2` | проб в вертикальном столбике сверху и столько же снизу |
| `SG_MIN_RUN` | `3` | мин. точек подряд одного цвета, чтобы считать блоком |
| `SG_BISECT_MAX` | `3` | макс. уточнений границы (переопределяется `?bisect=`) |
| `SG_BISECT_MIN_GAP` | `5` | px, дальше не мельчим при уточнении |

## Уроки прошлых итераций (учтены в оверлее)

- `NSWindow.sharingType = .none` — иначе `screencapture` захватывает
  собственные точки оверлея, и детектор находит их же на следующем скане
  (петля обратной связи).
- Без `isGeometryFlipped` — `y` инвертируется вручную
  (`(1 - normY) * height`), иначе сетка сдвигается на экране.
- `.rounded` bezel у `NSButton` игнорирует высоту фрейма (фиксированная
  Aqua-высота) — крупная кнопка сделана как плоский `CALayer`-фон
  (`isBordered = false`), а не системным bezel-стилем.
